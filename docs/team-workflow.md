# 团队工作方式说明

> 建立时间：2026-03-11
> 适用：每次开始新一轮工作前，直接按此流程推进

---

## 团队角色分工

| 角色 | 负责人 | 职责 |
|------|--------|------|
| 架构设计 & 验收 | Claude | 需求拆解、技术方案设计、代码审查、下一步规划 |
| 产品测试 & 沟通 | 用户（你） | 实际运行验证、效果预览、反馈给Claude、协调两侧 |
| 实际开发 | Codex | 按Claude的方案逐文件实现，不自行扩展范围 |

---

## 标准工作流程

```
1. Claude 出方案
   └── 明确目标、涉及文件、每步骤代码、验收标准、注意事项

2. 用户转达给 Codex
   └── 原文转发Claude的方案即可，Codex按方案执行

3. Codex 完成开发
   └── 回报改动位置（文件 + 行号）

4. 用户初步验收
   └── 实际启动应用，按验收清单逐条测试

5. 用户把 Codex 回报内容转给 Claude
   └── Claude 做最终代码审查（读源码，不是看描述）

6. Claude 给出结论
   └── 通过 → 给出下一个任务方案
   └── 有问题 → 指出具体行号和修复方式，退回 Codex

7. git commit & push（收尾）
   └── 每完成一个功能点就提交，保持记录清晰
```

---

## 方案格式规范（Claude输出）

每次给 Codex 的方案必须包含：

- **任务目标**：一句话说清楚做什么
- **涉及文件表**：文件名 + 操作类型（新增/修改）
- **分步骤代码**：每个文件改什么，给出完整代码片段
- **验收标准**：checkbox 列表，用户可逐项打勾
- **注意事项**：Codex 容易踩的坑

## 代码规范（Codex 必须遵守）

- 注释用中文
- 2空格缩进，单引号，分号结尾
- 所有 `await` 必须包 `try/catch`
- 禁止硬编码颜色，全部用 `var(--xxx)` CSS 变量
- 禁止用 `alert()` / `confirm()`，改用内联提示或二次点击确认
- 不要自行扩展需求范围，严格按方案执行
- 新增变量必须用 `let`/`const` 声明，禁止隐式全局

---

## 验收要点（Claude审查时关注）

1. 新增变量是否用 `let`/`const` 声明（防隐式全局）
2. innerHTML 拼接是否有 XSS 风险（内部数据可接受，外部输入必须转义）
3. 错误处理是否完整（try/catch + 友好提示）
4. 新增 IPC handler 是否有 `memorySystem` 空值保护
5. CSS 是否使用 CSS 变量（不硬编码颜色）
6. 事件监听是否正确挂载（DOMContentLoaded 内 vs 顶层）

# Bug专项团队协作方案

## 1. 团队角色（4人）

### A. 代码审查负责人（负责查看代码）
- 目标：快速定位问题点和风险点，给出可复现证据。
- 主要职责：
  - 审阅变更代码与相关模块调用链。
  - 复现 Bug，补充复现步骤、日志、截图。
  - 输出问题单：严重级别、影响范围、疑似根因。
- 产出物：
  - `Bug Report`（复现步骤 + 期望结果 + 实际结果 + 影响范围）。
  - `Code Review Findings`（按严重级别排序）。

### B. 修复负责人（负责修复 Bug）
- 目标：在不引入回归的前提下完成修复。
- 主要职责：
  - 根据问题单完成最小可行修复。
  - 补充单元测试/集成测试（至少覆盖本次缺陷场景）。
  - 提交 PR 并附验证结果。
- 产出物：
  - 修复代码与测试代码。
  - `Fix Note`（根因、修复点、潜在影响）。

### C. 架构负责人（负责把控框架）
- 目标：保证修复不破坏系统边界和长期可维护性。
- 主要职责：
  - 评审修复方案是否符合当前架构分层。
  - 审查跨模块依赖、IPC边界、数据模型变更风险。
  - 决策“临时补丁”还是“结构性修复”。
- 产出物：
  - `Architecture Review`（通过/驳回 + 原因 + 调整建议）。

### D. 设计审美负责人（负责设计和审美）
- 目标：保证修复后交互一致、视觉可用、体验不降级。
- 主要职责：
  - 验证 UI 改动是否符合既有风格（颜色、字体、间距、动效）。
  - 检查多窗口/多分辨率下视觉一致性。
  - 评审文案与交互反馈是否清晰。
- 产出物：
  - `UI/UX Check`（通过项、问题项、建议项）。

## 2. 执行流程（固定顺序）

1. 代码审查负责人建问题单（含复现证据）。
2. 架构负责人判定修复策略（补丁/重构）。
3. 修复负责人提交修复与测试。
4. 设计审美负责人做 UI/UX 回归检查。
5. 四方会签后合并。

## 3. 缺陷等级与处理时限

- P0（阻断）：2小时内给临时止血方案，24小时内完成正式修复。
- P1（高优先）：当日修复并回归。
- P2（中优先）：2个工作日内修复。
- P3（低优先）：进入迭代排期。

## 4. 每日协作节奏

- 每天 10:00：15 分钟 Bug Standup（昨日进展/今日计划/阻塞）。
- 每天 18:00：状态同步（新增 Bug、已修复、待验证、回归风险）。

## 5. 交付门禁（必须全部满足）

- 必须有复现步骤与修复前后对比证据。
- 必须有至少 1 条自动化测试或可重复手工测试清单。
- 架构负责人评审通过。
- 设计审美负责人评审通过。

## 一、创意概述

### 1.1 核心洞察

当前市场上的"思考模式"分为：
- **普通模式**：直接回答，不展示推理过程
- **深度思考**（DeepSeek R1、Claude Extended Thinking）：展示 CoT 推理链，更慢但更准

**缺失的第三种：类人思考模式** — 不是思考"答案是什么"，而是思考"我怎么才能做到"。

### 1.2 问题定义

当前 Agent 的思维模式是**工具匹配**：
```
用户请求 → 扫描 tools 数组 → 有匹配工具 → 调用
                              → 无匹配工具 → "我做不到"
```

人类的思维模式是**资源发现+方案合成**：
```
用户请求 → 我会吗？
           ├── 会 → 直接做
           └── 不会 → 我能学会吗？
                      ├── 搜一下怎么做
                      ├── 问问有经验的人
                      ├── 找找有没有现成工具
                      ├── 能不能用现有能力组合解决？
                      └── 尝试 → 失败 → 换方案 → 再试
```

### 1.3 类人思考的五步模型

```
┌──────────────────────────────────────────────────────────┐
│                    类人思考五步模型                         │
│                                                          │
│  Step 1: 能力自查（Introspection）                        │
│  "我的 Skills 库里有没有能直接解决的？"                      │
│  → 扫描 skill-registry → 语义匹配而非精确匹配               │
│                                                          │
│  Step 2: 记忆检索（Recall）                               │
│  "我以前遇到过类似问题吗？当时怎么解决的？"                   │
│  → 查询 memory_facts + memory_chunks（已有 FSRS 系统）      │
│                                                          │
│  Step 3: 资源发现（Discovery）                            │
│  "Web/GitHub 上有没有现成的解决方案？"                       │
│  → web_search + web_fetch（已有技能）                      │
│  → 提取关键方法/命令/代码片段                               │
│                                                          │
│  Step 4: 方案合成（Synthesis）                            │
│  "用我现有的能力（bash_run/file_write/...）                │
│   加上刚发现的方法，能组合出解决方案吗？"                     │
│  → 将 Step 1-3 的信息综合，生成执行计划                     │
│                                                          │
│  Step 5: 执行与反思（Execute & Reflect）                  │
│  "执行方案 → 成功？记住这个方法。失败？换方案重试"            │
│  → 成功路径写入记忆（下次直接用）                            │
│  → 失败路径也记录（避免重复错误）                            │
└──────────────────────────────────────────────────────────┘
```

## 二、与市场产品的对比

| 特性 | 普通模式 | 深度思考 (R1/o1) | **类人思考（我们的）** |
|------|----------|-------------------|----------------------|
| 核心能力 | 快速回答 | 推理链更准确 | 自主找到解决办法 |
| 面对"不会"时 | 直接说不会 | 推理后说不会（更有理由） | **想办法学会再做** |
| 使用工具 | 有就用，没有就算 | 有就用，没有推理替代 | **没有就去找/学/造** |
| 记忆参与 | 无 | 无 | **查历史经验 + 写入新经验** |
| 外部搜索 | 仅按指令搜索 | 仅按指令搜索 | **主动搜索解决方案** |
| 失败处理 | 报错停止 | 换思路推理 | **换方案+记住教训** |
| 技术壁垒 | 无 | 大模型能力 | **独立记忆+Skills+搜索** |

### 关键差异化

**深度思考解决的是「想不到」—— 类人思考解决的是「做不到」**

- DeepSeek R1 遇到数学题会深度推理找到解法 → 它在"想"
- 我们的宠物遇到"创建 Excel"会搜索方法、组合工具去做 → 它在"行动"

这两者不冲突，甚至可以叠加：深度推理确定"做什么" + 类人思考确定"怎么做"。

## 三、真实场景分析

### 场景 1：创建 Excel（用户提出的例子）

```
当前行为：
  用户："在桌面创建一个记账的 Excel"
  Agent：没有 create_excel 工具 → "抱歉，我目前没有创建 Excel 的能力"

类人思考：
  Step 1 能力自查：file_write ✅ bash_run ✅ → 可以写文件和执行命令
  Step 2 记忆检索：以前没创建过 Excel → 无经验
  Step 3 资源发现：搜索 "powershell create xlsx"
    → 找到 ImportExcel 模块 或 COM 对象方案
    → 也找到 "python openpyxl" 方案
  Step 4 方案合成：
    方案A: bash_run + PowerShell COM → 不需要额外安装
    方案B: bash_run + pip install openpyxl + python 脚本 → 更灵活
    → 选择方案A（零依赖）
  Step 5 执行：
    bash_run("powershell -Command \"...创建 Excel...\"") → 成功
    → 记忆写入：{"task":"创建Excel", "solution":"PowerShell COM", "success":true}
    → 下次再问直接用记忆中的方案
```

### 场景 2：批量重命名照片

```
当前行为：
  用户："把桌面上的照片按日期重命名"
  Agent：有 file_list + bash_run → 能做但策略简单，容易出错

类人思考：
  Step 1：file_list ✅ bash_run ✅ file_edit ✅
  Step 2：记忆中没有批量重命名经验
  Step 3：搜索 "batch rename photos by date powershell/python"
    → 找到 exiftool 方案（读取 EXIF 日期）
    → 找到 PowerShell Get-ItemProperty 方案
  Step 4：
    先 file_list 获取文件列表
    → 用 PowerShell 读取每张照片的修改日期
    → 按 "YYYY-MM-DD_原名" 格式重命名
    → 先用 ask_user 确认方案
  Step 5：执行 + 记忆
```

### 场景 3：发送邮件

```
当前行为：
  用户："帮我发一封邮件给 xxx"
  Agent："我没有发邮件的功能"

类人思考：
  Step 1：bash_run ✅ open_url ✅ → 理论上可以
  Step 2：记忆中无邮件经验
  Step 3：搜索方案
    → 方案A: PowerShell Send-MailMessage（需 SMTP 配置）
    → 方案B: open_url("mailto:xxx@xxx.com?subject=...") → 打开默认邮箱客户端
    → 方案C: Python smtplib
  Step 4：方案B 最简单且不需要用户配置密码
    → ask_user 确认收件人和内容
  Step 5：open_url("mailto:...") → 打开邮箱客户端，用户点发送即可
    → 记忆写入
```

## 四、技术可行性评估

### 4.1 实现路径

**核心改动：在 agent-runtime.js 的 system prompt 中注入「类人思考协议」**

这不是一个新的代码模块，而是一种**提示工程 + 工具编排策略**。

```
实现方式对比：

方案A: 纯提示工程（改 system prompt）
  - 在 system prompt 中注入"五步思考协议"
  - 引导 LLM 在回答"做不到"之前先搜索和尝试
  - 改动量：极小（只改 prompt）
  - 风险：LLM 可能不严格遵循
  - 推荐度：★★★★★（先验证）

方案B: 预处理管道（新增思考引擎）
  - 在用户消息到达 LLM 之前，先运行一个"能力评估"步骤
  - 自动搜索 Skills 库、记忆库、Web
  - 将搜索结果注入到 LLM 的上下文中
  - 改动量：中（新增 thinking-engine.js）
  - 风险：增加延迟、增加 token 消耗
  - 推荐度：★★★★（方案A 验证成功后升级）

方案C: 多 Agent 协作
  - 拆分为 Planner Agent + Researcher Agent + Executor Agent
  - Planner 分析任务，Researcher 搜索方案，Executor 执行
  - 改动量：大（需要多 Agent 调度框架）
  - 风险：复杂度高、延迟大
  - 推荐度：★★★（远期目标）
```

### 4.2 方案 A 详细设计（推荐首选）

**核心：在 system prompt 中注入"Never Say Can't"协议**

```javascript
// agent-runtime.js _buildMessages() 中注入
const HUMAN_THINKING_PROTOCOL = `
【类人思考协议 — 遇到问题时的思考方式】

当用户请求你做某件事时，在说"我做不到"之前，必须完成以下思考：

1. 能力自查：检查你的工具列表，有没有能组合使用的工具？
   - file_write 可以创建任何文本格式文件
   - bash_run 可以执行 PowerShell/Python/任何命令行程序
   - web_search + web_fetch 可以搜索任何技术方案
   - open_url 可以打开任何程序或网页
   - 这些工具的组合能力远超单个工具

2. 记忆检索：你以前解决过类似问题吗？如果有，直接用上次的方案。

3. 搜索方案：如果以上都没有，用 web_search 搜索：
   - "[任务关键词] powershell command"
   - "[任务关键词] python script"
   - "[任务关键词] command line tool"

4. 组合方案：将搜索到的方法与你的工具结合，生成可执行方案。
   - 优先选择零依赖方案（系统自带命令）
   - 其次选择轻量依赖（pip install 单个包）
   - 必要时 ask_user 确认方案

5. 只有在以上所有步骤都尝试后仍然无法解决，才可以说"目前无法完成"，
   并说明你尝试了什么、为什么不行、未来怎样才能做到。

记住：你有 bash_run（可以执行任何命令）+ web_search（可以搜索任何方案），
这意味着理论上你能完成任何可以通过命令行完成的任务。
`;
```

**改动范围：**
- `main-process/agent-runtime.js` — `_buildMessages()` 中，task/code 意图时注入协议
- 可能需要调整 `MAX_TOOL_ROUNDS_BY_INTENT.task` 从 15 到 20（思考需要更多轮次）

### 4.3 方案 B 详细设计（进阶版）

```
用户消息 → ThinkingEngine.preprocess()
           ├── Step 1: 意图理解 + 任务分解
           ├── Step 2: 能力扫描
           │   └── skillRegistry.matchBySemantics(task)  // 语义匹配，非精确匹配
           ├── Step 3: 记忆检索
           │   └── memorySystem.searchSimilarTasks(task)  // 找类似任务的解决方案
           ├── Step 4: 生成上下文注入
           │   └── { relevantSkills, pastSolutions, suggestedApproach }
           └── 注入到 messages → LLM 调用

新文件：main-process/thinking-engine.js
- preprocessTask(userMessage, skillRegistry, memorySystem)
- 返回 { contextInjection, estimatedComplexity, suggestedTools }
```

| 维度 | 方案 A (Prompt) | 方案 B (预处理) | 方案 C (多 Agent) |
|------|----------------|----------------|-------------------|

| 推荐实施顺序 | **第一步** | **第二步** | 远期 |

### 5.2 与"自学习自进化"的关系

```
类人思考（在线）              自学习进化（离线）
─────────────                ─────────────
遇到问题 → 即时搜索方案        定时学习 → 积累知识库
执行成功 → 写入记忆            评测团队 → 验证技能
下次遇到 → 直接用记忆          复习巩固 → 防止遗忘
         │                           │
         └────── 共享记忆系统 ─────────┘
         │                           │
         └── 在线思考的成功经验 ──────▶ 成为离线学习的种子

```

**这两个创意是一枚硬币的两面：**
- 类人思考 = **用时学**（遇到问题临时搜索解决）
- 自学习进化 = **闲时学**（主动储备知识技能）
- 两者通过 FSRS 记忆系统联通

## 六、技术壁垒分析

### 6.1 为什么竞品难以复制

| 壁垒 | 说明 |
|------|------|
| **独立记忆系统** | FSRS + SQLite + 向量搜索，本地运行，方案成功/失败经验持久化 |
| **Skills 生态** | 19 个内置 + 用户自定义 + 未来自学习生成的技能库 |
| **经验积累效应** | 用得越多，记忆越丰富，解决问题能力越强（网络效应） |
| **跨对话连续性** | 上周学会的方案，这周还能用（竞品每次对话重置） |
| **个性化方案选择** | 记住用户的环境（Windows/Mac、已装软件、偏好） |

### 6.2 护城河深度评估

```
竞品要复制这个功能，需要：
1. 本地持久化记忆系统（大多数竞品是云端无状态的）    → 6-8 周
2. Skills 执行框架（声明式定义+安全沙盒）           → 4-6 周
3. 经验学习闭环（成功/失败记录+检索+复用）          → 3-4 周
4. Web 搜索集成（DDG HTML 解析+降级链）             → 2-3 周
5. 用户交互审批（ask_user + approval humanizer）    → 1-2 周
────────────────────────────────────────────
总计：约 4-6 个月的全栈开发量

而我们已经有了 1-4 的基础，只需在 prompt 层面加入类人思考协议即可。
```

## 八、实施建议

### 8.1 分阶段路线图

**Phase 0: Prompt 验证（0.5 天）**
- 在 system prompt 中注入 HUMAN_THINKING_PROTOCOL
- 测试 5 个"当前做不到"的场景，观察 LLM 行为变化
- 目标：验证纯 prompt 方案的有效率

**Phase 1: Prompt 方案上线（1-2 天）**
- 正式注入类人思考协议到 task/code 意图的 system prompt
- 调整 MAX_TOOL_ROUNDS（可能需要增加，因为思考需要更多轮次）
- 记忆写入：成功的方案自动存入 memory_facts
- 记忆读取：下次类似任务自动检索历史方案

**Phase 2: 思考引擎（1-2 周）**
- 新建 thinking-engine.js 预处理管道
- skill-registry.js 增加语义搜索（不再是精确匹配）
- 思考过程可视化（chat-window 中显示"正在搜索方案..."）

**Phase 3: 经验社区（远期）**
- 用户 A 的宠物学会了"创建 Excel"的方法
- 方案上传到社区 → 用户 B 的宠物直接获得这个经验
- 形成"集体智慧"

# 创意评估：自学习自进化 Agent（仿生学习系统）

> 评估日期：2026-03-22
> 状态：创意评估阶段，待合适时机实施
> 关联系统：现有 FSRS 记忆系统（strength.js）、Skills 系统（18+1 内置技能）

---

## 一、创意概述

**核心愿景：** 让桌面宠物像人一样拥有「学习→巩固→遗忘→复习」的完整认知周期，能够自主从 Web 获取知识形成新技能，并通过自我评测团队验证学习效果。

**三层架构：**
1. **自主学习** — 定时从 Web 抓取资料，理解并转化为可执行的 SKILL.md 技能文件
2. **自我评测** — 评测团队自动验证新旧技能，检测遗忘、强化薄弱项
3. **仿生进化** — 模拟人类「学习→睡眠巩固→遗忘曲线→复习」的完整生命周期

**与现有系统的关系：**
- 已有 FSRS 动态强化系统（stability/retrievability/emotional_weight）→ 提供「遗忘曲线」基础
- 已有 Skills 系统（19 个内置技能 + SKILL.md 声明式定义）→ 提供「技能载体」
- 已有记忆分层（用户画像/重要记忆/对话历史）→ 提供「知识存储」
- **缺失的是：** 主动学习触发器、知识→技能转化器、自评测闭环

### 2.3 两篇必读论文深度分析

**论文 1：「Language Models Need Sleep」(OpenReview 2025)**

这篇论文提出了与你的创意高度吻合的"睡眠"范式：
- **清醒阶段**：模型正常处理用户请求（= 我们的日常对话）
- **睡眠阶段**：两步走
  - *记忆巩固（Knowledge Seeding）*：将短期记忆蒸馏到长期存储（= 我们的 FSRS reinforceMemory）
  - *做梦（Dreaming）*：模型用 RL 生成合成数据课程来复习和精炼（= 你说的「评测团队」）
- **关键发现**：不睡觉的模型在持续学习中性能急剧下降，"做梦"阶段对保持旧能力至关重要

**论文 2：「EvoSkill: Automated Skill Discovery」(arXiv:2603.02766)**

与你的「自己去 Web 找资料形成 skills」最对标：
- 运行进化循环：测试 → 识别失败模式 → 提出改进 → 评估 → 保留最优
- **零样本技能迁移**：在一个任务中学到的技能可直接用于未见过的任务
- Pareto 前沿选择：只保留真正提升性能的技能，不堆垃圾
- 实验结果：OfficeQA +7.3%, SealQA +12.1%

## 三、可行性评估

### 3.1 技术可行性：7.5/10（可行但有挑战）

| 组件 | 难度 | 可行性 | 说明 |
|------|------|--------|------|
| 定时学习调度器 | 低 | ✅ 完全可行 | 类似现有 reminder.js 的 cron 调度 |
| Web 知识获取 | 中 | ✅ 可行 | 已有 web_search + web_fetch 技能 |
| 知识→结构化提取 | 中 | ✅ 可行 | 已有 fact-extractor.js，可复用 |
| 知识→SKILL.md 转化 | **高** | ⚠️ 核心难点 | LLM 需要理解知识后生成可执行技能定义 |
| 技能沙盒测试 | 中 | ✅ 可行 | 可在隔离环境运行新技能验证 |
| 自我评测系统 | 中高 | ⚠️ 需设计 | 需要基准测试集 + 评分标准 |
| 记忆巩固（睡眠模式）| 中 | ✅ 可行 | FSRS 已有衰减框架，需增加主动巩固 |
| 遗忘检测与复习 | 低 | ✅ 可行 | 读取 retrievability 值即可判断 |

**核心难点分析：「知识→技能转化」**

这是整个系统最难的部分。人类学到的知识可以灵活运用，但 LLM Agent 需要将知识转化为结构化的可执行技能（SKILL.md + handler 代码）。

现实约束：
- 我们的 LLM 是调用 Qwen API，**不能微调模型权重**
- 所以"学习"本质上是**外部知识库的扩展**（类似 RAG），而非模型内化
- 技能转化需要 LLM 具备：理解文档 → 提取操作步骤 → 生成 SKILL.md YAML → 生成 handler 代码的能力

### 3.2 产品可行性：8/10（强差异化，但需谨慎定位）

**优势：**
- **市场独创性极强** — 目前没有任何桌面宠物/AI 助手产品具备"自主学习进化"能力
- **用户感知价值高** — "我的宠物昨天学会了一项新技能"是强情感钩子
- **与仿生人定位完美契合** — 这才是真正的"数字生命"，不是"AI 工具"
- **高级用户付费动力** — 定制学习方向、控制进化路径，是强付费功能
- **网络效应潜力** — 用户 A 的宠物学到的技能，可以分享给用户 B

**风险：**
- **学到垃圾技能** — Web 资料质量参差不齐，如果学到错误知识会严重损害信任
- **资源消耗** — 2 小时学习 = 大量 API 调用 = 成本问题
- **用户预期管理** — "自主学习"容易让用户期望过高，失望风险大
- **安全风险** — 自动生成的技能代码如果有 bug 或安全漏洞

### 3.3 商业可行性：7/10（高价值但需控制成本）

**成本模型估算（每用户每天）：**
```
2 小时学习会话：
  - Web 搜索：~20 次 × $0.001 = $0.02
  - 网页获取+解析：~20 次 × $0.002 = $0.04
  - LLM 知识提取：~20 次 × $0.005 = $0.10
  - LLM 技能生成：~5 次 × $0.01 = $0.05
  - 评测运行：~10 次 × $0.005 = $0.05
  ─────────────────────────────────
  总计：~$0.26/天/用户 ≈ ¥1.9/天 ≈ ¥57/月
```

这意味着：
- 免费用户**不可能**提供此功能
- 至少需要 Pro 套餐（¥79/月）才能覆盖成本
- 或者限制学习频率（每周 2 次而非每天）

## 四、建议架构设计

### 4.1 整体架构：「学习-睡眠-评测」三态循环

```
┌─────────────────────────────────────────────────┐
│                 用户配置层                        │
│  学习时间 / 学习方向 / 评测频率 / 进化速度        │
└──────────┬──────────────────────────┬────────────┘
           │                          │
    ┌──────▼──────┐           ┌───────▼───────┐
    │  清醒模式    │           │  睡眠模式      │
    │  (日常对话)  │ ──定时──▶ │  (空闲学习)    │
    │             │           │               │
    │  正常技能    │           │  ① 知识获取    │
    │  调用+执行   │           │  ② 知识提取    │
    │  记录交互    │           │  ③ 技能合成    │
    │             │           │  ④ 沙盒验证    │
    │  FSRS 衰减  │           │  ⑤ 记忆巩固    │
    └──────┬──────┘           └───────┬───────┘
           │                          │
           └──────────┬───────────────┘
                      │
              ┌───────▼───────┐
              │  评测模式      │
              │  (周期性)      │
              │               │
              │  ① 技能基准    │
              │  ② 遗忘检测    │
              │  ③ 回归测试    │
              │  ④ 进化报告    │
              └───────────────┘
```

### 4.2 学习引擎详细设计

```
学习会话流程（2 小时示例）：

Phase 1: 课程规划（10 分钟）
  ├── 读取用户配置的学习方向（如：Python 编程、健康养生、理财知识）
  ├── 扫描现有技能库，识别薄弱/空白领域（元认知）
  ├── 检查 FSRS 中 retrievability < 0.3 的旧知识（需要复习）
  └── 生成今日学习计划：70% 新知识 + 30% 复习旧知识

Phase 2: 知识获取（40 分钟）
  ├── Web 搜索目标主题（复用 web_search 技能）
  ├── 获取高质量页面内容（复用 web_fetch 技能）
  ├── LLM 提取结构化知识（复用 fact-extractor 框架）
  └── 存入知识库（memory_facts 表扩展）

Phase 3: 技能合成（30 分钟）
  ├── 分析知识是否可转化为可执行技能
  ├── 如果可以：生成 SKILL.md + handler 代码
  ├── 如果不能：作为对话知识存入记忆（丰富对话能力）
  └── 技能分级：draft（草稿）→ tested（已测试）→ verified（已验证）

Phase 4: 沙盒验证（30 分钟）
  ├── 在隔离环境运行新技能
  ├── 预设测试用例自动执行
  ├── 检查安全性（命令注入、文件访问越界等）
  └── 通过 → 标记为 tested；失败 → 标记为 draft 待改进

Phase 5: 记忆巩固（10 分钟）—— SleepGate 启发
  ├── 冲突检测：新知识是否与旧知识矛盾？
  ├── 遗忘门：过时/冲突的旧知识降低 stability
  ├── 摘要合并：相似知识合并为紧凑摘要
  └── 更新 FSRS 参数：新知识 S0=24h，复习的知识 S 增长
```

### 4.3 评测团队设计（Reflexion 启发）

```
评测团队组成（3 个 LLM Agent 角色）：

┌─────────────────┐
│  考官 (Examiner) │  生成测试题，覆盖新旧知识
├─────────────────┤
│  学生 (Student)  │  宠物本身，用当前知识作答
├─────────────────┤
│  评委 (Judge)    │  评分 + 生成语言反馈（Reflexion 风格）
└─────────────────┘

评测流程：
1. 考官根据技能库生成 N 道测试题（包含新学的和旧的）
2. 学生（宠物 Agent）执行测试题
3. 评委评分并给出反馈：
   - "技能 X 执行正确，但效率低于基线 → 建议优化"
   - "知识 Y 回答错误，上周还能答对 → 遗忘，需复习"
   - "新技能 Z 通过但有安全隐患 → 降级为 draft"
4. 反馈写入记忆（Reflexion 的语义梯度）
5. 下次学习时优先处理反馈中的薄弱项
```

## 七、实施建议

### 7.1 分阶段路线图

**Phase 0: 验证（1 周）— MVP 最小验证**
- 手动模拟一次"学习会话"：用现有 web_search → fact-extractor → 手写 SKILL.md
- 验证：LLM 能否从 Web 文章中生成合理的 SKILL.md？
- 目标：回答"知识→技能转化"这个核心假设是否成立

**Phase 1: 被动学习（2 周）**
- 从用户对话失败中学习（EvoSkill 模式）
- 用户问了宠物不会的事 → 记录 → 空闲时自动搜索学习 → 下次能回答
- 不需要定时调度，不需要评测团队
- 最小化改动，验证学习闭环

**Phase 2: 主动学习（3 周）**
- 定时学习调度器
- 用户配置学习方向
- 知识获取 + 记忆存储（不生成技能，只丰富对话知识库）
- SleepGate 式记忆巩固

**Phase 3: 技能合成（4 周）**
- 知识→SKILL.md 转化器
- 沙盒验证环境
- 技能生命周期管理（draft→tested→verified→dormant）

**Phase 4: 自我评测（2 周）**
- 评测团队（考官/学生/评委三角色）
- 遗忘检测 + 复习调度
- 进化报告生成（给用户看"你的宠物本周学会了..."）

### 7.2 推荐优先级

| 优先级 | 功能 | 理由 |
|--------|------|------|
| P0 | Phase 0 验证 | 不验证核心假设直接开发是最大风险 |
| P0 | Phase 1 被动学习 | 投入最小、用户感知最直接（"我的宠物变聪明了"） |
| P1 | Phase 2 主动学习 | 定时学习是创意的核心卖点 |
| P1 | Phase 3 技能合成 | 真正的"自主进化"，但技术风险高 |
| P2 | Phase 4 自我评测 | 锦上添花，Phase 1-3 稳定后再加 |
